Level generation pipeline. Decision (revised 2026-06-23): NO baking into per-level assets. One GameData-like ScriptableObject holds the xlsx reference and reads it directly. Single bootstrap scene + runtime generate. See [[prefer-single-so-over-baked-assets]].

**Data (Data asmdef, `Data.Levels` ns)** — `Assets/Scripts/Data/Levels/`:
- `CubeColor` (None/Yellow/Blue/Green/Red).
- `Level` (number, List<Vector2Int> splinePoints, List<Truck> trucks, `GridCenter()`) and `Truck` (id, gridPosition, rotationY, List<CubeColor> cubes) — plain in-memory classes, NOT assets.
- `XlsxReader`: dependency-free .xlsx parser (System.IO.Compression + System.Xml.Linq). Read(path/byte[]/Stream). Joins all <t> so multi-line cells keep `\n` (truck cell = "A\n90").
- `LevelParser.Parse(sheets)`: grid -> List<Level>. Splits levels by numeric markers in column A; integers in grid = ordered waypoints; letter cells = trucks (2nd line = Y rotation); grid column index = excelColumn - 2. Carriers row 2 = truck letters, columns below = cube colors top->bottom.
- `LevelData : SingletonScriptable<LevelData>`: the central SO (like `GameData`). Inspector field `excelFile` (UnityEngine.Object) — drag `Assets/_CaseStudy/Levels.xlsx`. `Levels`/`Get(i)`/`Count` read+cache via `ReadLevels()` -> resolves path (editor: AssetDatabase; build: StreamingAssets/<name>.xlsx) -> XlsxReader+LevelParser. Must live at `Resources/Scriptables/LevelData.asset` for `.Instance`. Has `Custom/Assets/Level Data` menu.

**Editor (Game.Editor asmdef, `Game.EditorTools` ns)** — `Assets/Scripts/Editor/LoopSort/`:
- `LevelDataEditor`: custom inspector with a "Read & Log Levels" button to validate parsing without play. (Old `LevelImporter` menu + per-level/`LevelDatabase` assets were DELETED.)

**Runtime (GamePlay asmdef, `Game.Level` ns)** — `Assets/Scripts/GamePlay/Level/`:
- `GridMapper.ToWorld(grid, center, cellSize)`: grid->world XZ; larger sheet row -> smaller Z.
- `LevelGenerator` (MonoBehaviour): `Generate(Level)` builds Dreamteck `SplineComputer` path (default Linear + Close, `cornerRadius`/`cornerSegments` quadratic-Bezier fillets = straight sides + rounded corners), builds the conveyor belt two ways via `useSplineMesh`: (a) DEFAULT `BuildSplineMesh` = Dreamteck `SplineMesh` continuous extrude (mesh+materials pulled from conveyorBlockPrefab, `meshRotation`/`meshOffset` to orient, autoCount) that bends through corners with no gaps — requires the belt fbx mesh Read/Write ON (Conveyor Block/End/End Thin .fbx metas set isReadable:1); (b) `PlaceConveyorBlocks` discrete blocks with `scaleToFit`+`blockOverlap` (miter gaps at corners, band-aid), lays conveyor segment prefabs along it (`PlaceConveyorBlocks`: steps by arc length via `Travel`, orients via `SplineSample.rotation`), instantiates trucks at pos/rot, lines up cubes along truck local -Z (cubeStackDirection, NOT vertical), assigns material per color. Serialized: cellSize, splineType, closeSpline, splineSampleRate, conveyorBlockPrefab, blockSpacing, blockRotationOffset, alignToSplineUp, truckPrefab, cubePrefab, cubeSize, cubeStackOrigin, cubeStackDirection, colorMaterials. Primitive-cube fallback if prefabs unset.
- `LevelManager` (MonoBehaviour): serialized `LevelData levelData` (falls back to `LevelData.Instance`) + `LevelGenerator`; loads by index, PlayerPrefs progress, `NextLevel()`/`Reload()`, fires `EventDispatcher.OnLevelChangedEvent`.

**asmdef wiring:** GamePlay references Data (c6db8b98...), Core (0e1e9ead...), Dreamteck.Splines (2fdef4a8...), Dreamteck.Utilities (fcf26afa...). Game.Editor references Data.

**Setup:** create LevelData asset at Resources/Scriptables/LevelData.asset, drag Levels.xlsx into it, click "Read & Log Levels" to verify, then wire LevelManager+LevelGenerator in the scene.

- `ConveyorBelt` (MonoBehaviour, Game.Levels): physics-based movement with REAL feel. Cubes keep gravity + free rotation; each FixedUpdate the belt projects the cube to the spline (`SplineComputer.Project`) and only nudges its along-tangent velocity toward `speed` via `AddForce(Acceleration)` — leaving gravity, lateral motion, spin and collisions to the solver so cubes tumble, hit belt walls/corners and queue. `Configure(spline,speed,acceleration,direction)`, `Add/Remove`, static `Prepare(GameObject)` (gravity ON, no constraints, continuous collision, BoxCollider if none). Code-based containment (`Contain`): clamps each box's lateral offset to `halfWidth` and height to `maxHeight` from the spline (invisible walls/ceiling), because the generated belt mesh walls are too short to keep boxes in. Configured via LevelGenerator `beltHalfWidth`/`beltMaxHeight`. The belt mesh gets a `MeshCollider` (SplineMesh path; per-block MeshCollider in block path) so cubes rest on it and bump corners. LevelGenerator adds the belt to the Conveyor GO, exposes `Belt`; fields conveyorSpeed/conveyorAcceleration/conveyorDirection. Debug `spawnTestCubesOnBelt`/`testCubeCount`/`testSpawnHeight` drops loose cubes to preview flow (motion only in Play). Truck cubes stay static.

**Gameplay (stack/unstack + sort), added 2026-06-23.** Model: top-match-or-empty accept + PERMANENT origin lock (no home/target color — that approach felt wrong: a truck with a matching top should accept). Consolidation comes from two rules: (a) TEMPORARY origin lock — a cube can't re-dock into its source truck until it travels away (prevents instant snap-back); (b) `HasCollector(color)` — if some truck already collects a color (top matches + has room), boxes won't seed a SECOND pile in an empty truck, so a color funnels into one truck instead of splitting across empties. Tap = unstack the whole top color-run to the belt; belt cube auto-docks into the first truck it passes (other than its origin) that `Accepts` it. Verified solvable on all 3 levels.
- Data class renamed `Truck` -> `TruckData` (collision with the new `Truck` MonoBehaviour).
- `Cube` (Game.Levels): `CubeColor color` + `originTruck` (the source truck; NEVER cleared, so the cube can't return there).
- `Truck` (Game.Levels, MonoBehaviour): capacity (fixed 4), dockPoint (nearest belt point via spline.Project), stack, `TopColor`; **`Accepts(color) = HasSpace && (empty || TopColor==color)`**; `IsUniform`, `IsComplete = full && IsUniform`; `Configure/AddCube/PopTop`. Tap via raycast in LevelController (truck has a TRIGGER BoxCollider — raycastable, doesn't block belt cubes; stack cubes' colliders also trigger).
- `Unstack` releases the entire top same-color run (staggered spawn heights), each tagged originTruck.
- Load/unload exclusivity: a truck being unloaded (`_unstacking`) is skipped by the dock loop (no loading while unloading); a truck being loaded (`_loading[t] > 0`, in-flight settle boxes) can't be tapped to unload. `_loading` reset on new level.
- Edge: tapping an already-complete truck when no other empty/matching truck exists can strand cubes (permanent lock) — acceptable user-error case for the prototype.
- `LevelGenerator`: `truckCapacity` fixed (default 4). Builds trucks as `Truck` (no color marker, no target assignment); exposes `Trucks`, public `CreateStackCube(color)` and `SpawnBeltCube(color,pos)`.
- `LevelController` (Game.Levels): reads the current level LIVE from its `generator` ref (no Setup wiring needed — only the `generator` field must be assigned; it detects new levels via `generator.Current`). Tap = raycast (Camera.main fallback) -> `GetComponentInParent<Truck>` -> `PopTop` -> `SpawnBeltCube` at dockPoint (cube.originTruck set so it can't snap back). FixedUpdate: belt cube docks into the first matching NON-origin truck within `dockRadius` (HORIZONTAL/XZ distance — cubes ride above the dock point so 3D distance never matched); origin lock clears once the cube leaves origin's radius. Dock = remove from belt, destroy, `CreateStackCube`+`AddCube`. `CheckWin`: belt empty + every truck is EITHER empty OR `IsComplete` (full of its single target color) -> `OnLevelCompleteEvent`. Truck capacity fixed (4).

**16-box cells (Phase 1 done 2026-06-23):** logic still cell-based but each Excel cell renders as 16 boxes (2x2x4). `Truck` stores boxes via `SlotLocalPosition`: a cell = cellWidth(4,X) × cellDepth(2,Z) × cellHeight(2,Y) = 16 boxes (fill X centered, then Z, then Y bottom-up); successive cells march along -Z with `cellGap` between them. capacity = cellCapacity(4) × 16 = 64; `Accepts = HasSpace && (empty || TopColor == color)` — pure top-color match, NO cell-boundary exception (a blue truck must not accept red just because it hit a 16 boundary). `boxesPerCell` is only for layout/capacity now. LevelGenerator config: cellCapacity/cellWidth/cellDepth/cellHeight/boxSize/stackOrigin; BuildTruck expands each cell color into boxesPerCell boxes. **Phase 2 DONE: animated per-box transfer** (coroutines, no DOTween dep). Unstack = `UnstackRoutine` releases the top color run one box at a time (`unstackInterval`); each box flies (`MoveWorld` smoothstep, `transferDuration`) from its truck slot to the belt launch point (`dockPoint + up*unstackHeight`) then `ActivateBeltCube` gives it physics. Boxes spin during transfer (`MoveSpin`, `flightSpin`): launch spins freely then gets `landTumble` angular velocity on the belt; settle spins then eases upright into the slot. Dock = logic instant (AddCube), then the new stack box flies from the belt into its slot (`SettleRoutine`). `_pendingTransfers` counts in-flight boxes (launch+settle); CheckWin defers while `_pendingTransfers > 0`, so win waits until every box finishes animating. New level does `StopAllCoroutines()` + resets the counter. Generator split: `CreateBeltCube` (no physics) + `ActivateBeltCube` + `SpawnBeltCube` (immediate, for debug). `_unstacking` HashSet guards re-tapping a truck mid-animation. boxSize must match the box prefab scale.

**Win screen + progression:** UIManager (UI asmdef) shows a BasePanel on `OnLevelComplete`; Next button -> `OnClickNextLevel` -> LevelManager.NextLevel(). DONE (needs scene wiring).

- `CameraFitter` (Game.Levels, on the Camera): keeps the camera's authored rotation, re-fits to level bounds (renderer bounds of `generator.Current`) on each new level (LateUpdate detects Current change). Projects the AABB onto the camera's right/up/forward axes for a tight fit. Orthographic: sets `orthographicSize = max(halfUp, halfRight/aspect)*padding` (distance only avoids clipping). Perspective: distance from FOV. `padding`/`minDistance` are plain floats. `Fit()` context menu.

**Refactor 2026-06-23 (split LevelGenerator):** LevelGenerator was ~400 lines; split into components in `Game.Levels`:
- `ConveyorBuilder` (component): builds the spline + SplineMesh + ConveyorBelt. `Build(splinePoints, center, cellSize, parent) -> ConveyorBelt`. Spline/mesh/corner params are now `const` (SplineKind=Linear, SplineSampleRate=20, CornerRadius=4, CornerSegments=6, closed always). The discrete-block conveyor path (PlaceConveyorBlocks/MeasureLength/ScaleAlong/Axis enum + all block fields) was DELETED (useSplineMesh was always true). meshRotation/meshOffset removed (channel defaults).
- `CubeFactory` (component): cube pooling + color materials + create/despawn. `CreateStackCube(color)`, `CreateBeltCube(color,pos,parent)`, `Despawn(cube)`. Holds `_stackCubePrefab/_beltCubePrefab/_colorMaterials/_stackPoolSize/_beltPoolSize`.
- `ColorMaterial` struct moved to its own file (`ColorMaterial.cs`).
- `ConveyorBelt` gained `Spline` getter and `Place(GameObject)` (= Add(Prepare(go))).
- `LevelGenerator` is now a slim orchestrator (~95 lines): refs `_conveyorBuilder`/`_cubeFactory`/`_truckPrefab`, serialized `_boxSize/_cellGap/_stackOrigin`, cell layout as `const` (CellSize=1, CellCapacity=4, CellWidth=4, CellDepth=2, CellHeight=2). `Generate`/`Clear`/`BuildTruck`; exposes `Current/Belt/Trucks/Cubes`. Debug test-cube spawner removed.
- LevelController now uses `generator.Cubes.*` and `generator.Belt.Place(...)` instead of LevelGenerator forwarding methods.
- SCENE WIRING required: add CubeFactory + ConveyorBuilder components; reassign prefabs/materials/conveyorBlock onto them (old LevelGenerator serialized refs are lost).

**Cube pooling (Core ObjectPooler):** cubes come from `ObjectPooler` pools instead of Instantiate/Destroy. LevelGenerator creates two pools (`_stackPoolKey`/`_beltPoolKey`, sizes `stackPoolSize`/`beltPoolSize`) lazily on first spawn (play mode only; edit-mode falls back to Instantiate). `SpawnCube` gets from pool, `Despawn` returns via `PutPoolObject` (uses `Cube.poolKey`). `Truck.PopTop` now RETURNS the Cube (no destroy); caller despawns it. Dock despawns the belt cube. `Clear` returns all `Cube`s under `_current` to their pools before destroying the level. Belt cubes are set `isKinematic=true` during the launch tween; `ConveyorBelt.Prepare` re-enables physics and resets velocity/angularVelocity (needed for reused pooled bodies).
